OpenAI Codex CLI — one protocol, five frontends, three sandboxes

Rust workspace of ~120 crates. Same agent core serves TUI, headless exec, IDE, app-server, and inbound MCP — through one Op/EventMsg protocol. Three-layer OS sandboxes per platform.

Language Rust License Apache 2.0 Workspace codex-rs/ ~120 crates ↗ openai/codex

TL;DR

One protocol, five frontends. A single Op / EventMsg protocol connects TUI, exec, IDE plugin, JSON-RPC app-server, and inbound MCP server to one agent core via submission_loop.
Sandboxing is the differentiator. Three layers per platform — Seatbelt + execpolicy + approval (macOS), bubblewrap + seccomp + Landlock + execpolicy (Linux), AppContainer + Job objects (Windows). .git, .codex, AGENTS.md are read-only even inside writable roots.
Claude-Code-compatible hooks. Same 8 events over JSON-over-stdio. The engine type is literally named ClaudeHooksEngine. Stop hooks can demand continuation.

Analogy: Codex CLI is the agent for people who want a real sandbox. Most agents say "trust the model"; Codex makes the OS say "I will not let the model do that".

2 · Submission Loop & Per-Turn Loop

sequenceDiagram participant F as Frontend participant S as submission_loop participant T as Task participant R as run_turn participant Hooks as Hooks engine participant LLM as Provider F->>S: Submission { id, op: UserTurn{...} } S->>T: spawn Task T->>R: run_turn(...) R->>R: pre-compact if near limit R->>R: resolve skills + plugin mentions + MCP tools R->>Hooks: SessionStart, UserPromptSubmit loop until done R->>R: drain pending input R->>LLM: run_sampling_request (Responses API) LLM-->>R: streamed events alt ContextWindowExceeded R->>R: run_auto_compact mid-turn else needs_follow_up R->>R: continue loop else done R->>Hooks: Stop, AfterAgent alt Stop hook returns continuation_fragment R->>R: re-enter loop with new user message end end end R-->>F: EventMsg stream

Five distinct re-entry points in run_turn (~600 lines) make the lifecycle subtle but powerful.

Seccomp modes — Restricted: deny connect/accept/bind/listen for non-AF_UNIX; always deny ptrace, process_vm_readv/writev, io_uring_*. ProxyRouted: invert — only AF_INET/AF_INET6 allowed (child reaches in-namespace TCP proxy).

4 · The Three-Layer Sandbox — macOS

# sandboxing/src/seatbelt.rs:602::create_seatbelt_command_args
/usr/bin/sandbox-exec \
  -p <policy> \
  -D READABLE_ROOT_0=/path/to/root \
  -- <command>

The hard-coded /usr/bin/sandbox-exec path is a real defense-in-depth choice — if $PATH is compromised, the system binary still runs. Final policy = base (closed-by-default) + per-root rules + (optional) network policy + restricted defaults.

Critical invariant across all platforms: PROTECTED_METADATA_PATH_NAMES — .git, .codex, .agents, AGENTS.md are read-only even inside writable roots. The agent can't rewrite its own rules.

5 · Approval & Execpolicy

flowchart LR A[shell command] --> B{execpolicy?} B -- forbidden --> Stop[reject] B -- allow --> C{AskForApproval policy?} B -- prompt --> Approve[ask user] C -- UnlessTrusted --> D{is_safe_command?} D -- yes --> Run[sandbox + run] D -- no --> Approve C -- OnRequest --> Run C -- Granular --> Splits[per-action booleans] C -- Never --> Stop Approve -- approved --> Run Approve -- denied --> Stop

Starlark execpolicy rules ([prefix_rule, host_executable]) gate commands before approval. Three lines of defense before a destructive command executes: execpolicy → user approval → sandbox.

10 · Deep Dive: Harness × Context × Memory

Codex CLI is built for one thing above all: keep an unsupervised LLM from doing damage. The harness, context, and memory all reflect that constraint.

10.1 · Harness engineering — Op/EventMsg protocol + submission loop + per-turn loop

Codex's harness is a Rust async runtime structured around a single Op/EventMsg protocol. Every frontend (TUI, headless exec, IDE plugin, JSON-RPC app-server, inbound MCP server) speaks the same protocol to the same core. The submission_loop dispatches ops to spawned Tasks; run_turn handles one full LLM round-trip with compaction, hooks, and tool dispatch.

10.3 · Memory management — AGENTS.md hierarchy + skills + session resume

11 · Strengths & Tradeoffs

💪

Strengths

  • Strongest sandbox among coding-agent CLIs
  • Claude-Code-compatible hooks (portable behavior)
  • One protocol, many frontends
  • First-class sub-agent infra w/ mailboxes
  • Mid-turn compaction
  • Hard-coded sandbox-exec for PATH defense
⚖️

Tradeoffs

  • run_turn ~600 lines, 5 re-entry points
  • Sandbox spawn overhead per shell call
  • Provider-aware compaction is leaky abstraction
  • Rust workspace (~120 crates) onboarding cost
  • Cross-session memory crate still in development

12 · Key Takeaways

  1. The sandbox is the lesson. Three independent enforcement layers + metadata masking = strongest current take on "untrust the model".
  2. One protocol unlocks many frontends. Every UI is just a different mouth feeding the same submission_loop.
  3. Hooks as continuation triggers. Stop hooks can demand the agent keep working — QA gate, not just audit log.
  4. Mid-turn compaction with model-aware re-injection. Leaky but pragmatically necessary.
  5. Skills as user fragments, not system messages. Preserves prompt cache while still adding capability.
