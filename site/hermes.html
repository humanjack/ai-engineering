Hermes Agent — The self-improving maximalist harness

One giant AIAgent class. 80+ tools. 7 terminal backends. 20+ messaging gateways. A curator that promotes the agent's own skills. The Swiss Army knife of agents.

Language Python 3.11+ License MIT Default model Claude Opus 4.6 ↗ NousResearch/hermes-agent

TL;DR

Monolithic by design. One AIAgent class — run_agent.py is ~16,400 LOC. Conventional OpenAI-style JSON tool-calling loop with every operational feature bolted on.
Self-improving via a curated skill loop. skill_manage lets the agent author its own markdown skills during a session; a background curator on the auxiliary model promotes / archives them on a 7-day cadence.
"Hermes" is branding, not coupling. Default model is anthropic/claude-opus-4.6. The runtime warns if you point it at real Hermes-3/4 chat models.

Analogy: Hermes Agent is the Swiss Army knife. Every blade is real, every blade has been used in production, and the whole thing is somehow still one piece of metal.

3 · The Self-Improving Skill Loop

stateDiagram-v2 [*] --> Created: agent calls skill_manage Created --> Active: written to ~/.hermes/skills/<name>/ Active --> Pinned: agent or user marks pinned Active --> Stale: 30 days unused Stale --> Active: agent uses it again Stale --> Archived: 90 days unused Pinned --> Pinned: immune to auto-transitions Archived --> Active: user restores

The curator (agent/curator.py, 1,781 LOC) runs every 7 days on the auxiliary model (to preserve the main model's prompt cache) and can: promote skills the agent uses repeatedly to pinned, archive unused skills, consolidate near-duplicates, patch skill bodies for clarity. Never auto-deletes. Pinned bypasses everything.

7 · Cache Discipline — The Frozen-Snapshot Trick

  1. System prompt built once per session, reused verbatim, stored in _cached_system_prompt
  2. New plugin context per turn injected into the user message, never the system prompt
  3. Memory writes visible to the writer's memory tool response (live state) but only baked into the system prompt on next session
  4. Curator runs on auxiliary model so its expensive reviews don't pollute main cache

Small choices that compound into a real cost-per-turn delta in long sessions.

8 · Testing & Trajectory Pipeline

  • ~17,000 tests across ~900 files
  • Canonical runner: scripts/run_tests.sh with -n 4 xdist, TZ=UTC, PYTHONHASHSEED=0
  • Heavy edge-case coverage: long context overflow, 4xx compression handling, Anthropic prompt-cache policy, dedup
  • mini_swe_runner.py (~28k LOC) — SWE-bench-style runner using Hermes's terminal backends
  • Trajectory format on disk = XML (<tool_call>/<tool_response>); wire format = JSON. Dual representation reflects Hermes 3 training heritage.

9 · Deep Dive: Harness × Context × Memory

Hermes Agent's distinctive engineering choices show up most clearly in how it manages the harness loop, the context bytes, and the persistent memory. Here's the breakdown.

9.1 · Harness engineering — one AIAgent class, five front doors, one IterationBudget

Hermes' harness is a single Python class. Five different entry points (CLI, Gateway, ACP, batch, MCP server, cron) all converge on one AIAgent instance that owns a synchronous while loop. The IterationBudget is shared between parent and subagents so the whole job has one cap.

9.2 · Context engineering — frozen system prompt, preflight + reactive compression

Hermes builds its system prompt once per session and keeps it byte-identical across turns so provider caches stay warm. New per-turn context goes into the user message instead. Compression is two-stage: a preflight pass before the call, plus a reactive fallback on context-overflow errors.

  • System prompt is frozen for the whole session — built once at start, persisted in state.db, reused byte-for-byte across turns
  • Per-turn context (plugin injections, scratchpad fragments) is appended to the user message, never the system prompt — so prefix caches stay valid

  • Defensive XML stripping — open-model leakage of <tool_call> tags inside prose is stripped at run_agent.py:3706-3798; wire format is JSON-only

9.3 · Memory management — frozen snapshot + bounded files + one external provider

Memory in Hermes is intentionally bounded and curated. The system prompt sees a frozen snapshot of MEMORY.md / USER.md at session start; durable on-disk writes update files but not the prompt; an optional external provider (Honcho, Mem0, etc.) can run in parallel — but only one at a time.

  • Frozen snapshot pattern — tools/memory_tool.py:1-24 documents the contract: prompt is stable, tool response shows live state, writes are durable but visible only to next session
  • Bounded character budget — MEMORY.md / USER.md are length-capped; agent must curate, not just append

  • Background curator — runs on the auxiliary model every 7 days; protects the main model's cache by never touching it

10 · Strengths & Tradeoffs

💪

Strengths

  • Everything in the box
  • Self-improving skill loop is a real differentiator
  • Disciplined cache management
  • 17k tests, very long-tail edge case coverage
⚖️

Tradeoffs

  • 16k-LOC run_agent.py is a commitment
  • Python plugins are unsandboxed
  • New hook events require core PRs
  • Cron coupled to gateway uptime

11 · Key Takeaways

  1. Engineering > Novelty. "Shipping an agent" is mostly retries, surrogates, JSON repair, dead-connection cleanup, and provider quirks.
  2. Cache discipline pays — frozen-snapshot system prompts + user-message context injection + auxiliary-model curator.
  3. AST-based plugin discovery beats hand-maintained registries.
  4. Self-improvement needs invariants — never auto-delete, pinned bypasses, archive recoverable.
  5. Bounded memory is honest — explicit curation > infinite vector recall.
