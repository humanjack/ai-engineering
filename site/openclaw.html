OpenClaw — A multi-channel productization of Pi

A local-first personal AI assistant daemon. Embeds Pi as its agent runtime; layers on 22+ messaging channels, 122 plugins, 56 skills, cron, hooks, MCP, and disciplined prompt caching.

Language TypeScript (ESM) License MIT Embeds @earendil-works/pi-coding-agent v0.74 ↗ openclaw/openclaw

TL;DR

OpenClaw is Pi, productized. The agent loop is literally Pi's — OpenClaw imports createAgentSession from @earendil-works/pi-coding-agent and embeds it.
One Gateway per host. A long-lived local daemon with a WebSocket control plane. Many clients (CLI, menu bar, mobile, web UI) connect to it.
Everything Pi doesn't ship — cron, hooks, webhooks, MCP server + client, Docker/SSH sandboxes, file-based memory, channel routing — is added as plugins under extensions/.

Analogy: If Pi is the V8 engine, OpenClaw is the entire car — body, dashboard, infotainment, and routes to 22 highway systems.

5 · Prompt Cache Discipline

OpenClaw treats prompt caching as a first-class engineering invariant. From AGENTS.md:

"Prompt cache: deterministic ordering for maps/sets/registries/plugin lists/files/network results before model/tool payloads. Preserve old transcript bytes when possible."

This shows up as:

  • Custom transcript entry type openclaw.cache-ttl marking TTL anchors inside JSONL sessions
  • isCacheTtlEligibleProvider() deciding per-provider whether to emit TTL markers (Anthropic family, Kilocode-routed Anthropic, Google/Gemini)
  • Cache-TTL-aware pruning in pi-hooks/context-pruning.ts
  • Pre-compaction reminder: "save important notes to memory before compacting"

8 · Deep Dive: Harness × Context × Memory

OpenClaw inherits Pi's loop but builds a much larger surrounding system. Here's how the three engineering layers fit together.

8.1 · Harness engineering — one Gateway, many surfaces

OpenClaw's harness is a long-lived WebSocket Gateway that owns a single embedded Pi loop per session. Multiple clients (CLI, menu bar, mobile, web UI) and 22+ messaging channels all talk to that one Gateway. Concurrency is serialised through per-session command lanes.

  • Per-session lanes — concurrent inbound events for the same session are serialised; different sessions run in parallel
  • Two hook layers — Gateway-level internal hooks for system events (message:received, session:compact:before) and plugin-level loop hooks tied to the embedded Pi loop
  • Auth profile rotation — each session can rotate between provider auth profiles with automatic failover on rate limits

D2.2 · The single wrapper does 5 things

  • 1. Fire before_tool_call plugin hook (pi-tools.before-tool-call.ts:95-170) — can mutate params in place

  • 5. Inject channel delivery context — channel, to, threadId, senderIsOwner

9.2 · Specialties

  • Provider plugins, not provider code. Each provider is its own folder under extensions/<name>/ with an openclaw.plugin.json manifest declaring auth choices, endpoint claims, capability contracts, model catalog, and per-model normalisation rules
  • Auth profile rotation + failover. A single persona can list multiple auth profiles (e.g. 3 Anthropic API keys + 1 Claude Code OAuth). On 429 / 5xx, OpenClaw rotates to the next profile within the same provider and retries — no model swap, no context loss
  • Per-provider cache-TTL eligibility. isCacheTtlEligibleProvider(provider, modelId, modelApi) in src/agents/pi-embedded-runner/cache-ttl.ts decides whether the session should emit openclaw.cache-ttl transcript markers. Currently: Anthropic family, Kilocode-routed Anthropic, Google/Gemini
  • Plugin-overridable eligibility. A provider plugin can return a custom verdict via resolveProviderCacheTtlEligibility() — useful for self-hosted gateways
  • Deterministic prompt ordering for cache stability. AGENTS.md:47 mandates that maps / sets / plugin registries serialise in a stable order so the prompt prefix stays byte-identical across turns — keeps every provider's prefix cache warm
  • Provider plugins ship per-model normalisation — e.g. the Anthropic plugin can rewrite claude-3-opus requests differently from claude-sonnet-4-6 without touching pi-ai
  • OpenClaw can be its own MCP server — openclaw mcp serve exposes channel conversations as MCP tools to OTHER LLMs (Claude Code, Cursor, etc.), making the agent itself a context source

9.3 · Cache-TTL Eligibility Snippet

// src/agents/pi-embedded-runner/cache-ttl.ts
export function isCacheTtlEligibleProvider(
  provider: string,
  modelId: string,
  modelApi?: string,
): boolean {
  const pluginEligibility = resolveProviderCacheTtlEligibility({...});
  if (pluginEligibility !== undefined) return pluginEligibility;
  return (
    isAnthropicFamilyCacheTtlEligible({ provider, modelId, modelApi }) ||
    (provider === "kilocode" && isAnthropicModelRef(modelId)) ||
    isGooglePromptCacheEligible({ modelApi, modelId })
  );
}

Together with the deterministic ordering rule, this is OpenClaw's "operational layer above Pi-ai" — Pi handles the bytes-over-the-wire; OpenClaw handles which key to use, when to mark a cache boundary, and how to fail over when one provider misbehaves.

10 · Strengths & Tradeoffs

💪

Strengths

  • 22+ messaging channels from one daemon
  • Disciplined prompt-cache engineering
  • Plugin SDK with enforced boundaries
  • Pi embedding gives full event-stream control
  • Two well-defined hook levels
⚖️

Tradeoffs

  • Tight coupling to Pi's SDK surface
  • Two hook layers is a learning curve
  • Code plugin vs bundle plugin distinction
  • Large surface area (8,948 production TS files)
