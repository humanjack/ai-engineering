OpenCode — The LSP-native coding agent

A TypeScript server + Go TUI in one binary. Edits wait for compiler diagnostics and feed them back to the model. Tree-sitter bash permissions. Shadow-git snapshots. No vector DB.

Server TypeScript / Bun TUI (v1) Go / Bubble Tea Pinned to sst/opencode@v1.0.0 ↗ sst/opencode

TL;DR

Two cooperating processes, one binary. TS Hono server (Bun) owns the agent loop. Go Bubble Tea TUI is embedded into the binary, extracted on first launch, Bun.spawn'd as a child connected over localhost HTTP/SSE.
LSP is a model-feedback channel. After every edit, the tool synchronously waits up to 3s for diagnostics and embeds compiler errors into the tool result. The model sees the type errors and self-corrects next step.
Three distinctive engineering bets: shadow-git snapshots for free revertability, tree-sitter bash permissions for per-AST-node safety, and ripgrep-only retrieval (no embeddings).

Analogy: OpenCode is the agent that pair-programs with your language server. Where most agents treat code as text, OpenCode treats it as a tree that the compiler can talk back about.

2 · The Cleverest Detail — Inverse RPC Over Long-Poll

HTTP doesn't push, so OpenCode does inverse RPC over long-poll in packages/opencode/src/server/tui.ts:1-69:

const request = new AsyncQueue<TuiRequest>()
const response = new AsyncQueue<any>()

export async function callTui(ctx: Context) {
  const body = await ctx.req.json()
  request.push({ path: ctx.req.path, body })
  return response.next()  // blocks until TUI POSTs back
}

The TUI sits in a loop hitting GET /tui/control/next. When the server queues work, the next GET drains it; the TUI processes and POSTs to /tui/control/response. Bidirectional comms without WebSockets.

4 · LSP as Model Feedback — The Defining Behavior

In packages/opencode/src/tool/edit.ts:104-116:

await LSP.touchFile(filePath, /* waitForDiagnostics */ true)
const diagnostics = await LSP.diagnostics()
const errors = diagnostics
  .filter(d => d.severity === 1)
  .map(d => `${d.range.start.line}: ${d.message}`)
  .join("\n")
return { output: `${editSummary}\n\nDiagnostics:\n${errors}`, ... }

After every edit, the tool blocks up to 3s for the language server to reply, then embeds compiler errors directly into the tool result. The model sees "unused import 'foo'" immediately and can fix it on the next step without being asked.

Plus two LSP tools the model can invoke directly:

  • lsp-diagnostics — proactively fetch all diagnostics
  • lsp-hover — ask for type info at a location

5 · Tree-Sitter Bash Permissions

Naïve regex bash perms break on pipes, subshells, and string boundaries. OpenCode uses web-tree-sitter with the bash grammar to extract every command node, then matches each literal command string against the agent's wildcard set:

const tree = await parser().then(p => p.parse(params.command))
const permissions = await Agent.get(ctx.agent).then(x => x.permission.bash)
for (const node of tree.rootNode.descendantsOfType("command")) {
  // ... check each command against agent's wildcard rules
}

For rg pattern | grep todo | xargs rm -rf, each of the four commands is checked independently. Catches escapes regex misses.

6 · Shadow-Git Snapshots

Every session can revert via a shadow git repository:

GIT_DIR=$XDG_DATA/opencode/snapshot/$projectID
GIT_WORK_TREE=$worktree

Every assistant turn commits to the shadow repo; Snapshot.patch(hash) reverts. Zero new infrastructure — just clever git usage. (Tied to git; non-git projects skip snapshots.)

10 · Deep Dive: Harness × Context × Memory

OpenCode's distinguishing engineering trick is using the compiler itself as a feedback channel. The harness, context, and memory all bend around that goal.

10.1 · Harness engineering — TS server owns the loop, Go TUI is just a renderer

OpenCode ships one binary that contains two processes. The TS Hono server owns all logic and state; the Go Bubble Tea TUI is a thin renderer connected by HTTP + SSE + a long-poll inverse-RPC channel.

  • Explicit loop — the AI SDK is set to stopWhen: stepCountIs(1) so the server can run snapshots, plugin hooks, and tool dispatches between every step (session/prompt.ts:225-427)
  • Inverse RPC — server pushes work to TUI via long-poll GET /tui/control/next (no WebSockets needed)
  • Per-edit shadow snapshot — every edit creates a git-style snapshot in a side worktree; revert is one tool call
  • Tree-sitter bash permissions — bash command strings are parsed; every command AST node is matched against agent's wildcard rules separately, so git status; rm -rf / doesn't sneak through
  • 5 plugin hooks — chat.params (mutate request), tool.use.before, tool.use.after, chunk, session.end

10.2 · Context engineering — LSP as a context source, no embeddings

OpenCode's context model is unusual: no vector DB, no embeddings, no semantic recall. Instead, the agent uses ripgrep + LSP as its primary retrieval tools. After each edit, the LSP synchronously feeds compiler diagnostics back into the next context window.

10.3 · Memory management — JSON on disk + shadow git, no semantic layer

OpenCode's memory is intentionally minimalist. Sessions are JSON files; durable knowledge is the project's AGENTS.md (which the agent can edit); revertable state lives in a shadow git worktree per session.

  • Session = JSON file — easy to inspect, diff, version-control; SQLite migration is in progress on the dev branch
  • Compacted ranges — instead of mutating history, OpenCode marks ranges as compacted and filters them on prompt build; the original is preserved for replay
  • AGENTS.md is durable memory — the agent can Edit it directly; persists across sessions and projects
  • Shadow git — every edit creates a snapshot in a side worktree, making revert a single git operation; no in-memory undo stack
  • No semantic memory — by design. ripgrep + LSP do the heavy lifting; the philosophical bet is "code is structured, not vibey"

11 · Strengths & Tradeoffs

💪

Strengths

  • LSP-as-feedback is genuinely novel
  • Tree-sitter bash perms beat regex
  • Shadow-git = free revertability
  • OpenAPI → many clients
  • Provider-tuned system prompts
⚖️

Tradeoffs

  • Two-language build/release complexity
  • Long-poll control channel is single-tenant
  • JSON storage doesn't scale (SQLite migration in progress)
  • 5 hook surfaces is a lot
  • Plugins are unsandboxed Bun imports
