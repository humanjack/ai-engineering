OpenHands — event-sourced agent with a FastAPI sandbox

A three-repo polystack. The flagship is FastAPI + React; the agent loop lives in agent-sdk; the sandbox runs its own ~40-router FastAPI server with VS Code inside Docker. Every action and observation is an immutable event.

Language Python 62% · TS 36% License MIT (core) · Polyform (enterprise) Latest 1.7.0 (May 2026) SWE-Bench Verified 77.6 ↗ All-Hands-AI/OpenHands

TL;DR

Event-sourcing as ground truth. Every action and observation is a frozen Pydantic event in a persisted EventLog. Replay, fork(), rerun_actions(), and encrypted persistence all fall out of one design choice — most other agents hold an ephemeral message array.
The sandbox is its own FastAPI app. openhands-agent-server runs inside the Docker container with ~40 routers (terminal, file, git, VS Code, bash, desktop, hooks, MCP). It's a PyInstaller binary the host SDK talks to over HTTP/WS — not a thin docker exec wrapper.
Local/remote symmetry via one factory. Conversation.__new__() inspects the workspace and returns either LocalConversation or RemoteConversation — same API, remote variant proxies every call to the in-sandbox server. Laptop ↔ cloud is one argument.

Analogy: OpenHands is the agent built like a distributed system. Events are immutable, the sandbox is a service, and Local/Remote are interchangeable transports. Where Codex makes the OS enforce, OpenHands makes the architecture enforce.

3 · Event-Sourcing as Ground Truth

The conversation is a log of immutable events, not a list of messages. Every event subclasses Event (event/base.py:16); transcript-participating events subclass LLMConvertibleEvent with an abstract to_llm_message().

EventPurpose
MessageEventUser or assistant text turn
ActionEventOne tool call. Fields: thought, reasoning_content, tool_name, tool_call_id, security_risk, critic_result, summary
ObservationEventTool result; action_id links back
SystemPromptEventSystem prompt versioning

What this buys:

  • Fork — LocalConversation.fork() snapshots and replays the event log into a new state
  • Rerun — rerun_actions() re-executes a slice without re-prompting the model
  • Encrypted persistence — ConversationState accepts a _cipher; logs at rest are encrypted
  • Discriminated polymorphism — Events, Agents (agent_kind: "openhands" | "acp"), Workspaces, Sandboxes all (de)serialize via Pydantic's DiscriminatedUnionMixin

Same call site, same returned API. What changes is the transport. RemoteConversation proxies every call to the in-sandbox openhands-agent-server; rather than maintain two control loops, OpenHands runs one and lets the workspace differ.

6 · The In-Sandbox Agent-Server

flowchart LR Host[Host: RemoteConversation] subgraph Container["Docker / KVM / Remote VM"] API[api.py FastAPI app] subgraph Routers["~20 routers"] R1[conversation_router] R2[bash_router] R3[file_router] R4[git_router] R5[mcp_router] R6[vscode_router] R7[hooks_router] R8[skills_router] R9[event_router] R10[desktop_router] end subgraph Services["services"] S1[bash_service] S2[conversation_service] S3[event_service] S4[hooks_service] S5[tool_preload_service] S6[vscode_service] S7[desktop_service] end Persist[persistence/] Sock[sockets — websockets] end Host -.HTTP REST.-> API Host -.WS.-> Sock API --> Routers Routers --> Services Services --> Persist

Compiled to a single PyInstaller binary (agent-server.spec) and installed into the Docker image. Three sandbox backends in app_server/sandbox/: docker (default, KVM in 1.7), process (no isolation, trusted dev), remote (cloud).

Why a server inside the sandbox? (1) VS Code lives there — vscode_router + extensions are shipped, the host tunnels in. (2) Tool execution is uniform — local and remote conversations talk to the same FastAPI interface. (3) ACP variant: conversation_router_acp lets external Anthropic-style agents drive it.

8 · Context Management — Two Orthogonal Components

Condenser — "the context is too big"

Default is LLMSummarizingCondenser (condenser/llm_summarizing_condenser.py, 282 LOC). Algorithm: find forgettable window → LLM-summarize → replace with one summary event, preserving the first keep_first for prompt-cache hits.

FieldDefaultRole
max_size240 SDK / 80 presetTrigger threshold
keep_first2 SDK / 4 presetCache-stable prefix
minimum_progress0.1Don't bother if we'd evict too little
hard_context_reset_max_retries5Fallback for runaway context

Stuck Detector — "the agent is in a loop"

stuck_detector.py (429 LOC) detects five patterns: repeating A→O pairs, action+repeated errors, agent monologue (no tools for N turns), A-B-A-B alternation, context-window errors despite condensation. On match → status STUCK, run() exits.

13 · Strengths & Tradeoffs

💪

Strengths

  • Event-sourced state → replay, fork, encrypted persistence for free
  • In-sandbox FastAPI server with VS Code, hooks, MCP inside the box
  • Five-tier productization off one core (SDK → CLI → GUI → Cloud → Enterprise)
  • ~50 providers via LiteLLM with fallback chains + OAuth subscription login
  • First-class MCP client + server on host and inside sandbox
  • 77.6 SWE-Bench Verified; 6-benchmark eval suite
⚖️

Tradeoffs

  • v1.0 split scatters code across 7+ repos — real onboarding work
  • Heavyweight stack: FastAPI host + FastAPI sandbox + React + Docker
  • enterprise/ is Polyform (30-day annual non-commercial trial)
  • Agent-server PyInstaller binary adds HTTP hop + build artifact
  • Old OpenDevin docs rank high but cite paths that 404 in v1
  • One Agent class — no subclass diversity

14 · Key Takeaways

  1. Events are the data model. Replay, fork, encryption, and remote symmetry are all consequences of treating the conversation as an immutable log rather than a message array.
  2. The sandbox is a service, not a shell. A full FastAPI agent-server inside Docker gets you VS Code, MCP, and hooks inside the box for free.
  3. One agent, four extension axes. Tools, presets, skills (YAML), subagent registry — replacing the old agent-class hierarchy. CodeActAgent is now a YAML string.
  4. Condenser + stuck detector are orthogonal. "Context is too big" vs "agent is in a loop" — kept separable.
  5. The v1.0 split is the story. OpenDevin was a monolith; OpenHands is a polyrepo where each layer has one job.
