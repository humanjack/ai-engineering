Overmind — the agent that optimizes other agents

A policy-driven optimizer that ingests your existing LLM agent, runs it under a synthesized eval suite, diagnoses failures with OpenTelemetry tracing, and uses an internal coding agent to rewrite prompts, tool descriptions, and pipeline logic — promoting changes only past regression-aware acceptance gates.

Language Python 3.10–3.13 License MIT (pyproject classifier) Status Alpha · v0.1.49 Maintainer Overmind Ltd ↗ overmind-core/overmind

Identity check

This is NOT the well-known DarthSim/overmind Procfile process manager. overmind-core/overmind is an autonomous LLM-agent optimizer — a tool that ingests a Python (or JS) agent and iteratively rewrites its prompts, tool descriptions, model choices, and pipeline logic to improve measured performance. Peer tools: DSPy, GEPA, TextGrad, OptiLLM, Trace. Not a coding agent like Claude Code / Codex / OpenCode — but it uses an internal coding agent to do its edits, and it integrates into those coding agents via Agent Skills.

TL;DR

Optimizer, not chatbot. The outer loop is best-of-N candidate generation driven by an evaluation score, not a tool-call loop driven by user intent. Each candidate is a parallel git worktree.
Coding agent inside a coding agent. Overmind's inner Claude-style agent (coding_agent/agent.py, ~180 LOC, MAX_STEPS=50, doom-loop guard) applies plans in worktrees. Overmind is also invokable from Claude Code / Cursor via three Agent Skills.
Policy-driven. Optimization is anchored to a domain policies.md + a strict eval_spec.json (weights must sum to 100) + a dataset.json. No policy → no signal → no optimization.
Regression-aware acceptance. A candidate is promoted only if it passes byte-equality on read_only_paths, the data-leakage detector, the 4-tier score threshold, the complexity penalty (capped at 60% of raw gain), and a holdout probe.

The inner coding agent uses the same patterns that Claude Code / Codex / OpenCode use externally: a tight tool-call loop, file tracker with stale-read detection, doom-loop guard. FileTracker enforces read-before-write (raises FileNotReadError) and compares mtime + size on every edit (raises FileStaleError).

This is the most distinctive piece. Where most agents accept whatever the model outputs, Overmind treats every candidate edit as a scientific claim that must clear five independent filters. The system prompt for the inner coding agent explicitly forbids hardcoding test-input responses, and the optimizer's data-leakage detector cross-checks candidate code against training expected outputs.

Output types are deliberately constrained to text | enum | number | boolean. This keeps both the LLM-as-judge layer cheap and the mechanical layer decisive. The "weights sum to exactly 100 via formula-driven allocation" rule is enforced as a hard validation gate during spec generation — no fuzzy weighting.

Provenance-aware optimization. Every span carries one of six SourceTag levels (real subprocess → real LLM → real code → cassette replay → simulation → static-only). They aggregate into a numeric Confidence ∈ [0.0, 1.0] per candidate. Design principle from the source: "Overmind can still reason about an agent when it cannot fully execute it, but it must never pretend simulated signal is the same as real signal." LLM calls always run real, even in shadow mode — replaying LLMs would zero out the optimization signal.

7 · FailureRegistry — cross-run failure memory

stateDiagram-v2 [*] --> Open: new failure cluster
SHA-256(failing fields × tool patterns) Open --> Resolved: candidate passes this cluster Resolved --> Regressed: later candidate fails it again Regressed --> Resolved: subsequent candidate fixes it Resolved --> [*]: persists across runs
via run_state.json

Cases that share the same structural signature — failing output fields crossed with tool-call patterns — are clustered by SHA-256 hash. Clusters get a lifecycle (open → resolved → regressed) that persists in run_state.json across overmind optimize invocations. This turns "did we fix this last time?" into a first-class signal the analyzer can target.

10 · Security & reliability

Bash tool sandboxing~ subprocess.run(..., shell=True) with no chroot/container; default 120s timeout. Only a system-prompt guideline asks the agent to prefer dedicated tools.
Filesystem safety✓ FileTracker enforces read-before-write (FileNotReadError) and stale-read detection via mtime + size (FileStaleError). Per-path locks for thread safety.
Read-only enforcement✓ accept_step byte-equality-checks every read_only_path after candidate edits — any change ⇒ reject.
Worktree isolation✓ One git worktree per candidate ⇒ parallel edits never collide on disk.
SecretsSingle .overmind/.env file (no per-agent envs, deliberately, to avoid placeholder shadowing); skills enforce a hard "Mandatory Keys Pause" gate; key_ok() rejects <set-me>/empty patterns.
Anti-overfitting✓ Holdout probe · complexity penalty · data-leakage detector · system prompt forbidding test-input hardcoding · accept_step rejection of memorized examples.
Failure taxonomy11 modes: IMPORT_ERROR · MODULE_LEVEL_CRASH · MISSING_DEPENDENCY · API_KEY_MISSING · BROWSER_RUNTIME_ERROR · NETWORK_ERROR · TIMEOUT · SCHEMA_MISMATCH · CRASH · UNKNOWN · NONE
Retries / robustnessAnalyzer shrinks case counts on context overflow; LLM-judge retries with text-similarity fallback; OTel exporter has 10s flush timeout; subprocess timeouts default 120s; capped attempt/success lists at 50.
Trust modelDesigned for the user's own machine. No sandbox like Codex's Landlock/seccomp/Seatbelt — security is "you trust your own agent's source code."

11 · Evaluation, tests, CI

  • ~80+ pytest files under tests/ covering CLI, configs, analyzer, optimizer, policy/spec/dataset generators, evaluator, tracing (test_tracer.py, test_spans.py, test_tracing_lifecycle.py), execution backend, registry, bundling.
  • Make targets: make test (parallel, -n auto, worksteal dist), test-cov, test-serial, lint-check, lint-format. Uses uv run when available, falls back to python. test_spans.py is excluded from parallel runs (OTel global-state).
  • CI workflows:
    • ci.yml — lint-only on PR / push to main, uv + Python 3.13.
    • pr-checks.yml — pytest on Python 3.12 + a version-bump check that fails the PR if pyproject.toml version isn't strictly greater than main.
    • publish.yml — PyPI release on tag.
    • test_io_from_sdk.yml — SDK I/O contract tests.
  • Public benchmark via examples. The 8 baseline agents in examples/ double as Overmind's own benchmark suite with published before/after deltas:
    • contract_extractor: 74.3 → 94.1 (+19.8)
    • lead_qualifier: 52.5 → 73.9 (+21.4)
    Per the README: "Baselines are intentionally leaky (vague prompts, thin tool docstrings, wrong model tier, unbounded iteration)."

Explicit cross-run learning: seed_failed_attempts() and seed_successful_changes() warm-start the next overmind optimize invocation with the prior run's lessons. This is the single biggest difference from "session-scoped" agents like Claude Code or Codex.

13 · Tools & integration surfaces

Inner-agent tools

First-party Python only — no MCP/plugin layer. Tools live in overmind/coding_agent/tools/ and implement a ToolBase Protocol returning ToolResult{output, title, metadata}. The ToolContext carries session_id, worktree, cwd, and FileTracker hooks. Schemas convert to OpenAI function-call format via schema_to_openai().

ToolNotes
readRecords into FileTracker for later staleness checks
writeRequires prior read on existing files (FileNotReadError otherwise)
editmtime + size verified against last read; raises FileStaleError on drift
grep / globRead-only discovery
bashsubprocess.run(..., shell=True) in ctx.cwd; 120s default timeout; no allowlist
apply_patchUnified-diff application

Outer integration: 3 Agent Skills

Three SKILL.md folders in overmind/skills/ turn Overmind into a Claude-Code-loadable workflow: /overmind-register-agent, /overmind-generate-spec-and-dataset, /overmind-optimize-agent. Each skill carefully splits deterministic Python scripts (registry lookups, env validation) from coding-agent reasoning (schema inference, harness design).

Auto-instrumentation

The package depends on overmind-instrumentation-anthropic, overmind-instrumentation-openai, overmind-instrumentation-google-generativeai, overmind-instrumentation-agno, and opentelemetry-overmind — OTel auto-instrumentation for the major LLM SDKs.

15 · Strengths & tradeoffs

💪

Strengths

  • Real anti-overfitting machinery — 5 gates, holdout probe, leakage detector, complexity cap
  • Cross-run learning via FailureRegistry, not just per-session
  • Unified tracing — parent-span stitching across every subprocess
  • Provenance honesty — simulated signal ≠ real signal, by design
  • Composes with Claude Code / Cursor via 3 Skills
⚖️

Tradeoffs

  • Alpha — v0.1.49, ~12 stars, expect API churn
  • No top-level LICENSE file (only pyproject classifier)
  • Docs are placeholders — docs/agent-improvement.md and docs/tracing.md are 0 bytes
  • Hosted docs gated behind Cloudflare
  • Inner bash tool is permissive (shell=True, no allowlist) — fine for trusted local use, not for hostile inputs
  • No MCP — first-party Python tools only
  • Cost model — repeated agent runs across N candidates × K iterations gets expensive without cassettes

16 · Five takeaways

  1. Optimization is the new agent surface. Once you have a working agent, the next problem is "make it 20 points better on my eval." Overmind treats that as a first-class workflow.
  2. Acceptance gates > raw scores. A score going up is necessary, not sufficient. Regression-awareness, complexity penalty, and holdout probes prevent the easy wins from being illusions.
  3. Provenance matters. The 6-tier SourceTag hierarchy + Confidence ∈ [0.0, 1.0] is a clean answer to "how much should I trust this number?"
  4. Cross-run failure memory. Most agents forget between sessions. FailureRegistry persists clusters with a lifecycle — and the analyzer can target them.
  5. Coding agents are infrastructure. The inner coding agent here is ~180 LOC because the patterns (tool-call loop, FileTracker, doom-loop guard) are now well-understood. The action has moved up the stack — to what you point them at.
