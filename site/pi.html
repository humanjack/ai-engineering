Pi — A minimal, self-extending coding agent

4 built-in tools. Tree-shaped sessions. The agent writes its own extensions in TypeScript. Powers OpenClaw.

Language TypeScript License MIT Author Mario Zechner ↗ badlogic/pi-mono

TL;DR

Minimal core. Only Read, Write, Edit, Bash ship in the box. Everything else is added by the agent writing its own TypeScript extensions at runtime.
Sessions are trees. Every message has a parentId. Branch off to fix a tool, do a code review, rewind back to where you were — no context pollution.
Provider-agnostic. A single Context object is portable across Anthropic, OpenAI, Google, xAI, Mistral, and 15+ providers — swap models mid-conversation without losing state.

Analogy: Pi is the busybox of coding agents. Tiny core, hot-reloadable plugins, "if you want it to do X, ask it to write X."

1 · Why It Exists

Most agents grow by accretion: more tools, more prompts, more middleware, more lock-in. Pi inverts the design. The core does exactly four things; everything else is built by the agent itself (or by the user) as TypeScript extensions. The model — frontier models that already know TS — becomes the runtime's compiler.

The slogan: "clay, not concrete." The agent maintains its own functionality.

3 · The Heart: runLoop

The entire agent loop is a single function in packages/agent/src/agent-loop.ts:155:

async function runLoop(currentContext, newMessages, config, signal, emit, streamFn) {
  while (!signal?.aborted) {
    // Stream assistant response (line ~193)
    const message = await streamAssistantResponse(...);
    const toolCalls = message.content.filter(c => c.type === "toolCall");

    if (toolCalls.length === 0) break;   // No tool calls → done
    // Execute tools — sequential OR parallel depending on tool.kind (line ~381)
    await executeToolCalls(...);
  }
}

That's it. The complexity hides in events, tool execution semantics, and the steering system.

During the streaming + tool-execution phase the user can steer (interrupt with a new instruction) or queue follow-ups. Both are first-class state machine transitions, not hacks.

5 · Tree-Shaped Sessions

Sessions are JSONL files where every entry has id + parentId. This unlocks:

graph TD M1["msg-1: user — 'Fix login bug'"] M2["msg-2: assistant — tool calls"] M3["msg-3: tool result"] M4["msg-4: assistant — 'Done'"] M5["msg-5: user — 'Also fix logout'"] M6["msg-6: user — '/review' (side-quest)"] M7["msg-7: assistant — review findings"] M1 --> M2 --> M3 --> M4 M4 --> M5 M4 --> M6 --> M7
  • /tree — jump to any past point, continue from there
  • /fork — create a new session file from a branch point
  • Compaction — older messages get summarized; full tree preserved in JSONL
  • Side-quests — review/refactor in a branch, rewind, summarize the work back into main

8 · Deep Dive: Harness × Context × Memory

The three engineering disciplines that decide whether an agent feels solid or flaky. Here is Pi's stack for each:

8.1 · Harness engineering — event-driven, steerable, hot-reloadable

Pi's harness is unusually small but does three things well: it emits a typed event stream, lets the user steer mid-tool-call, and supports hot-reloading extensions so the agent can rewrite its own tools without restarting.

  • One typed event stream — every UI surface (TUI, Slack bot, web UI, OpenClaw) consumes the same AgentEvent enum from packages/agent/src/agent-loop.ts
  • Steering — mid-tool-call interrupt that skips remaining tool calls and re-prompts the assistant with new instructions
  • Follow-up queue — stack messages to run after the current turn finishes, without losing focus on the current task
  • Hot-reload — extensions are TS modules; edit-and-save reloads them in-place. The agent can write its own extension, save, then call it next turn
  • Custom message types — extensions can inject non-LLM messages (UI-only, metadata) that convertToLlm() filters out before the API call

8.2 · Context engineering — portable across providers, tree-shaped on disk

  • Provider-agnostic Context — { systemPrompt, messages[], tools[] } is serializable; a session built on Claude can resume on GPT-5 or Gemini with no state loss
  • Side-quest branches — /review spawns a child branch, does its work, summarizes back, then rewinds — the main thread never sees the noise

  • Token + cost tracking in pi-ai means context decisions can be cost-aware, not just length-aware

8.3 · Memory management — file-based, walked up the tree

Pi doesn't ship a vector DB. Memory is plain markdown files that the agent reads on session start and can edit during the session via the standard Write / Edit tools.

  • Walk-up resolution — Pi starts at cwd and walks up the filesystem looking for AGENTS.md / CLAUDE.md; closer files override farther ones
  • Global fallback — ~/.pi/AGENTS.md is appended last, so cross-project preferences (style guides, default models, etc.) stay sticky

Deep Dive II · Tool Use

Pi's tool system is the smallest in the series — and on purpose. Four built-ins, TypeBox schemas pumped straight to providers, default-parallel dispatch, no approval gate in the core. Everything else lives in extensions.

9 · LLM Integration — The pi-ai Layer

Pi rolls its own LLM SDK called pi-ai (published as @earendil-works/pi-ai). It is not a wrapper around LangChain, LiteLLM, or any other meta-SDK — it sits directly on top of each provider's official client and normalises them into a single streaming interface.

9.2 · Specialties

  • Provider-routed via model.api. The unified stream(model, context, options) in packages/ai/src/stream.ts calls getApiProvider(model.api) and delegates — no dynamic if (provider==='anthropic') branching
  • Official SDKs underneath, not raw HTTP. Each provider module imports the vendor SDK (e.g. import Anthropic from "@anthropic-ai/sdk") and translates Pi's internal Context shape to the SDK's request schema
  • 13 provider modules, 1 import. register-builtins.ts registers every provider at startup; agent code only ever sees stream() / complete() / streamSimple()
  • Two API shapes for OpenAI — openai-completions (legacy chat completions) AND openai-responses (newer Responses API) — plus openai-codex-responses for ChatGPT-backend Codex compatibility
  • Cache retention as first-class config. PI_CACHE_RETENTION=short|long|none selects Anthropic ephemeral-vs-1h cache control; the same model attribute is honoured everywhere
  • Hardened JSON parsing. parseJsonWithRepair, parseStreamingJson, and sanitizeSurrogates sit in packages/ai/src/utils/ for malformed tool-call arguments and bad UTF-16 from weak models
  • Cross-provider Context portability. Sessions store an LLM-agnostic Context — a Claude session can resume on GPT-5 because transform-messages.ts normalises the shape per-API at the boundary
  • Bedrock via Anthropic + AWS SDK. amazon-bedrock uses both @aws-sdk/client-bedrock-runtime AND patterns from the Anthropic SDK — same prompt cache discipline as native Anthropic

10 · Strengths & Tradeoffs

🧬

Strengths

  • Smallest cognitive footprint
  • Hot-reload = self-debugging agent
  • True provider neutrality
  • Tree sessions are a novel UX primitive
⚖️

Tradeoffs

  • No MCP by default (bridge via mcporter)
  • TS-first extension API
  • Less batteries-included than Deep Agents or OpenCode
  • TUI runtime tied to Bun

11 · When to Choose Pi

  • You want to own your agent's behavior end-to-end
  • You're comfortable in TypeScript and prefer an extension API to a plugin marketplace
  • You need to multiplex providers (e.g., Claude for code, Gemini for vision, GPT-5 for browsing)
  • You want tree-based session UX without writing it yourself
  • You want a stable runtime that powers other interfaces (Slack, web, Telegram → see OpenClaw)
